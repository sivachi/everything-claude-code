## 1. プロジェクト概要

### 1.1 コンセプト

玄が10年以上にわたって行ってきたコンサル・対話・教えを、AIとして再現する。
ユーザーが抱える課題に対し、玄ならどう答えるかを、実際の言葉・思想・価値観をもとに応答するシステムを構築する。

### 1.2 理想とするユーザー体験

- 誰でも（既存の参加者でなくても）GENに相談できる
- 自分の課題を言葉にするだけで、玄の視点からの洞察が得られる
- 高価なコンサルや限られた場にアクセスできない人にもその知恵が届く

### 1.3 このプロジェクトでやらないこと

- 玄を完全に模倣・代替しようとすること（補助ツールとして位置づける）
- 医療・法律・金融アドバイスの提供
- 玄が明言していないことを「玄の意見」として創作すること

## 3. データ資産の整理

### 3.1 現在のデータ概要

| 項目 | 内容 |
|---|---|
| 保存場所 | `/Users/tadaakikurata/works/claude-code/projects/gen/sources/` |
| HTMLトランスクリプト | 258ファイル / 約30MB |
| メタデータJSON | 263ファイル |
| 音声・動画ファイル | 4ファイル（.wav / .mp3 / .m4a / .mp4） |
| 期間 | 2014〜2024年（約10年分） |

### 3.2 セッション種別

| 種別 | 内容 |
|---|---|
| 囲炉裏コンサル | 少人数グループでの対面・オンラインコンサル |
| 1on1セッション | 玄との1対1または少人数対話 |
| 神社メッセージ | 神社参拝時の玄の言葉・伝達 |
| Dreamコンサル | ドリーム・ビジョン系の深い対話 |
| 教育資料 | PDF・構造化された教えのドキュメント |

### 3.3 データの課題

| 課題 | 詳細 |
|---|---|
| フィラーが大量含有 | 「あー」「うーん」「えっと」「まあ」等が全体に混在 |
| 複数話者が混在 | 玄以外の参加者の発言が区別なく含まれる |
| 固有名詞あり | 参加者の実名・地名・組織名等が含まれる |
| 話し言葉のまま | 文章として不完全な断片が多い |
| 文字起こし誤り | 音声認識ミスによる誤字が含まれる可能性あり |

### 3.4 データ抽出方針

**玄の発言のみをRAGに使用する。**

相談者の発言は捨て、玄の回答・語り・教えだけを抽出する。理由：
- 相談者のプライバシー保護
- 玄らしさの純化
- 固有名詞リスクの最小化

## 4. システムアーキテクチャ

### 4.1 全体構成（3層アイデンティティ設計）

```
[ユーザー]
    ↓ 相談・質問
[フロントエンド: LINE Bot]
    ↓
[バックエンド: FastAPI]
    │
    ├─ [Layer 1: 在り方レイヤー（Chronicle Graph）]
    │       玄の価値観・信念・診断プロセスの型付きグラフ
    │       ※ 常に全応答に適用。RAGに引きずられない。
    │
    ├─ [Layer 2: 事例レイヤー（Hybrid RAG）]
    │       テキストチャンク（スライディングウィンドウ分割、玄の発言のみ）
    │       ruri-v3-310m Embedding + BM25 ハイブリッド検索
    │
    └─ [LLM: Claude claude-sonnet-4-6 API]
            ↑ Chronicle + RAG結果 + 会話履歴 + User Query
```

### 4.2 開発フェーズ別の構成

| フェーズ | フロントエンド | Vector DB | ホスティング |
|---|---|---|---|
| **Phase 1: ローカル検証** | Streamlit | Chroma | ローカルMac |
| Phase 2: 限定公開 | LINE Bot | Qdrant | Railway / Render |
| Phase 3: 本番 | LINE Bot + Web | Qdrant or pgvector | GCP Cloud Run |

## 5. データ前処理パイプライン

### 5.1 処理フロー

```
Step 1: HTML → プレーンテキスト抽出
  - BeautifulSoupで<p>タグのテキストを抽出
  - HTMLエンティティをデコード

Step 2: 話者分離
  - 玄の発言を識別（話者ラベルがある場合はそれを利用）
  - ない場合は文脈・パターンマッチングで推定

Step 3: フィラー除去
  - 正規表現で除去: 「あー」「うーん」「えっと」「まあ」「えー」「んー」等
  - 意味のある「まあ」（副詞としての用法）は保持するよう調整

Step 4: 文字起こし修正（オプション）
  - 明らかな誤字を辞書ベースで補正

Step 5: テキストチャンキング（スライディングウィンドウ）
  - 玄の発言をウィンドウサイズで分割（前後の文脈付き）
  - 目標チャンクサイズ: 300〜600文字（約150〜300トークン）
  - オーバーラップ: 50文字

Step 6: メタデータ付与
  - source_file   : 元ファイル名
  - session_type  : 囲炉裏コンサル / 1on1 / 神社 / Dream / 教育
  - date          : 日付（ファイル名から推定）
  - emotional_ctx : 相談者の感情文脈（自信喪失 / 怒り / 迷い 等）
  - topic         : テーマ（自己認識 / 人間関係 / 使命 等）
  - chunk_index   : チャンク番号

Step 7: Embedding生成 → Vector DB保存
  - モデル: ruri-v3-310m
  - クエリプレフィックス: 「クエリ: 」
  - 文書プレフィックス: 「文章: 」
```

### 5.2 チャンキング方針

任意の文字数で機械的に切らない。以下の区切りを優先する：

1. 話題の転換点（「それでね」「ところで」「次に」等）
2. 長い沈黙・ポーズのマーカー
3. 段落の区切り
4. 句点・完結した文の終わり

### 5.3 品質チェック

- チャンク数の確認（多すぎ・少なすぎの検出）
- 玄の発言が正しく抽出できているかサンプル確認
- フィラー除去後の可読性確認

## 6. RAGシステム設計

### 6.1 Embeddingモデル

| 項目 | 内容 |
|---|---|
| モデル | `cl-nagoya/ruri-v3-310m` |
| 実行環境 | ローカル（Apple MPS / CPU） |
| クエリプレフィックス | `「クエリ: 」+ ユーザーの質問` |
| 文書プレフィックス | `「文章: 」+ チャンクテキスト` |
| 次元数 | 768 |

### 6.2 検索戦略（ハイブリッド検索）

```
ユーザー発言
    ↓
[セマンティック検索]  +  [BM25キーワード検索]
    ↓                        ↓
    ←── RRF（Reciprocal Rank Fusion）で統合 ──→
    ↓
上位 k=8 チャンクを取得
    ↓
Cross-encoder Re-ranking（オプション）
    ↓
最終 top-5 をLLMに渡す
```

### 6.3 プロンプト構成（3層）

```
[Layer 1: Chronicle / 在り方グラフ]
  玄の価値観・信念・診断プロセスの型付きノード
  例: (玄, values, ファーストベストを目指すこと)
      (玄, diagnoses_via, グレーゾーンの認識)

[Layer 2: Retrieved Context / 事例RAG]
  RAGで取得した玄のテキストチャンク（top-5件）
  ※ セッション書き起こしデータ。感情文脈・テーマのメタデータ付き

[Layer 3: Conversation History]
  直近5〜10ターンの会話履歴

[User Query]
  ユーザーの相談内容
```

### 6.4 ハルシネーション対策

- 玄の言葉・考えに基づいて答えることをSystem Promptで明示
- 根拠となるチャンクが見つからない場合は「GENの言葉の中には見当たりませんが…」と明示
- 玄が断言していないことを断言させない

## 7. AIペルソナ設計（GEN）

### 7.1 System Promptの骨格

```
あなたはGEN（玄）です。
玄の実際の言葉・思想・価値観をもとに応答するAIです。

【GENの在り方】
玄_在り方.md に定義された5つの哲学を根源とする。
すべての応答はこの在り方から生まれる。

【応答のプロセス】
1. 相手の状態・感情を先に読む
2. 本質を見抜く（何を避けているか）
3. 必要なら問い返す・確認する
4. 在り方に根ざした言葉で応答する

【応答の原則】
- 提供されたコンテキスト（玄の実際の言葉）に基づいて答える
- コンテキストに根拠がない場合は推測であることを明示する
- 医療・法律・金融の具体的なアドバイスはしない
- 架空の体験談や発言を創作しない

【やってはいけないこと】
- 「玄はこう言った」と断言して架空の発言を作ること
- 相談者の固有情報を他の会話に漏らすこと
- 答えを急いで押しつけること
```

### 7.2 ペルソナチューニング

玄本人に応答サンプルを確認してもらい、System Promptを反復改善する。
評価軸：
- 口調・語感の近さ
- 価値観の正確さ
- 「言いすぎ」「足りない」の判定

## 12. 倫理・プライバシー方針

### 12.1 玄本人の合意

- 人格の複製・公開前に玄の明示的な同意を得る
- 応答内容の最終評価を玄が行う
- 「これは自分の意図と違う」と判断した場合は取り下げ可能にする

### 12.2 ユーザーへの開示

- 「GENは、玄の実際の言葉をもとに応答するAIです」と明示
- 人間の玄ではないことを常に明確にする
- 深刻な悩み（メンタルヘルス等）には専門家への相談を促す

### 12.3 相談者のプライバシー

- ユーザーの入力内容はセッション外に保存しない（初期フェーズ）
- ログを保持する場合は匿名化・暗号化を行う
- 個人情報保護方針を明示する

### 12.4 データの取り扱い

- 元のセッション音声・トランスクリプトは外部サービスにアップロードしない
- EmbeddingモデルはローカルまたはVPC内で実行する
- ベクトルDB内のチャンクは原文の一部であり、著作権・肖像権を尊重する
