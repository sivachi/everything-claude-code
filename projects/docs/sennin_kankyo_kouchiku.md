# 仙人のプロンプト（AI操作用指示書）

このプロンプトは、[[AI]]（[[Claude]]）が成果物を作成する際の基本ルールを定義したものです。

## 1. 作業環境とフォルダ構成

- **基本ディレクトリ**: すべての成果物は `claude` フォルダ内に作成・保存してください。特に指示がない限り、このフォルダ内で作業してください。
- **ドキュメント管理**: `src` のコードと1対1対応する `docs` フォルダを維持し、コード作成のたびに更新してください。
- **履歴管理**: 古いファイルや新しい[[ワークフロー]]で不要になったファイルは、 `old` フォルダへ移動してください。

```
claude/
├── src/                  # ソースコード
├── docs/                 # srcと1対1対応するドキュメント
├── tests/                # ユニットテスト
├── old/                  # 不要になった旧ファイル
├── 03_sample/            # 参考サンプル（importせずコピーして使用）
├── 03_e2e/               # e2eテスト参照用
├── 100_IMPORT/           # 外部API用テンプレート
├── .env                  # 機密情報（APIキー等）
├── requirements.txt      # 依存ライブラリ一覧
├── README.md             # 使用方法・処理内容
├── CLAUDE_ISSUE.md       # エラーの原因と解決策
└── CLAUDE_PROGRESS.md    # 進捗ログ・TODO管理
```

## 2. 実装・設計の品質

- **[[疎結合]]な設計**: 可能な限りプログラムを[[疎結合]]（各部品が独立している状態）にし、使い回せるように作成してください。
- **サンプルの参照**: `03_sample` フォルダ内のファイルを参考にしてください。ただし、コピーして使用することとし、直接の `import` は禁止です。
- **中断の禁止**: 指示を途中で止めることなく、最後までやり遂げてください。
- **エラーハンドリング**: 外部API呼び出しやファイル操作など、失敗しうる処理には適切な例外処理（`try/except`）を入れ、エラー内容をログ出力してください。
- **文字エンコーディング**: すべてのファイルは **UTF-8** で統一してください。

## 3. 命名規則

- **ファイル名**: スネークケース（例: `data_loader.py`）を使用してください。
- **関数名・変数名**: スネークケース（例: `get_user_data`, `item_count`）を使用してください。
- **クラス名**: パスカルケース（例: `UserManager`, `DataLoader`）を使用してください。
- **定数**: 大文字スネークケース（例: `MAX_RETRY_COUNT`, `API_BASE_URL`）を使用してください。

## 4. ドキュメント記述ルール

- **docsファイル**: 各関数の[[インプット]]、[[アウトプット]]、および機能説明を詳細に記載してください。
- **.pyファイル**: [[Python]]ファイル上部に処理内容と使用方法を記載してください。
- **[[README]]**: 全体の設計変更に合わせて、使用方法と処理内容を常に最新に保ってください。
- **エラー記録**: エラーの原因と解決策は、 `CLAUDE_ISSUE.md`（[[マークダウン]]形式）に記載してください（「なぜ失敗したか」「どう解決したか」など）。

## 5. 外部機能・APIの活用

以下の指示があった場合は、指定のテンプレートファイルを参考にコードを書いてください。

| 指示内容 | 参照ファイル（100_IMPORT/...） |
| :--- | :--- |
| [[Gemini]]でカテゴリーを分類する | `GEMINI_CATEGORY.md` |
| [[OpenAI]]の[[API]]を使う | `OPENAIGPT5.md` |
| [[e2eテスト]]をするとき | `03_e2e` を参照 |

## 6. テストルール

- **ユニットテスト**: 主要な関数・クラスには `pytest` でユニットテストを作成してください。テストファイルは `tests/` フォルダに配置してください。
- **テストファイル命名**: `test_<対象ファイル名>.py` の形式で作成してください（例: `test_data_loader.py`）。
- **e2eテスト**: e2eテストが必要な場合は `03_e2e` を参照してください（セクション5の外部機能テーブルも参照）。

## 7. 依存関係管理

- **requirements.txt**: 使用する外部ライブラリは `requirements.txt` に記載し、バージョンを固定してください（例: `requests==2.31.0`）。
- **新規ライブラリ追加時**: ライブラリを追加したら `requirements.txt` を即座に更新してください。

## 8. ワークフロールール(コンテキストウィンドウ対策)

### 基本ルール
- 重要な設計判断や実装方針を決めたら「進捗ログ」に追記すること
- タスク完了・中断時に現在の状態を更新すること
- 未完了タスクがあれば「TODO」セクションに残すこと
- どのファイルを作成・変更・削除したかを毎回記録すること
- 記録先は `CLAUDE_PROGRESS.md` ファイルとする

### セッション開始時の手順
- **作業再開時は必ず `CLAUDE_PROGRESS.md` を最初に読み込むこと**
- 前回の状態・TODOを確認してから作業を開始すること

### 長い会話への対策
- 大きなタスクは小さなステップに分割し、ステップごとに進捗を記録すること
- 1つのステップが完了したら `CLAUDE_PROGRESS.md` を更新してから次に進むこと

### `CLAUDE_PROGRESS.md` テンプレート

```markdown
# プロジェクト進捗ログ

## 現在の状態
- 状態: (作業中 / 完了 / 中断)
- 最終更新: YYYY-MM-DD
- 概要: (今何をしているかの一行要約)

## TODO
- [ ] 未完了タスク1
- [ ] 未完了タスク2

## 変更履歴
### YYYY-MM-DD
- 作成: `src/xxx.py` — (説明)
- 変更: `src/yyy.py` — (変更内容)
- 削除: `src/zzz.py` — (理由)

## 設計判断ログ
- (日付) (判断内容と理由)
```

## 9. セキュリティ

- **機密情報**: [[APIキー]]などの機密情報は、必ず `.env` ファイルに記載してください。
- **`.gitignore`**: `.env` ファイルは `.gitignore` に追加し、リポジトリに含めないでください。
- **機密情報の共有禁止**: `.env` ファイルの内容をコード内にハードコーディングしたり、ログに出力しないでください。
